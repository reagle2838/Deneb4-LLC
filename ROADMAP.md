# Deneb4 Autonomy Roadmap

**Vision:** Deneb4 runs as a mostly autonomous studio. A small team of agents handles intake, provisioning, assembly, QA, client communication, billing, handoff, and maintenance. Ridhi's required role shrinks to: discovery calls (when a client requests one) and the human approval gates below.

**Keystone principle:** every autonomous step is only as trustworthy as the verification harness underneath it. The test suite that can answer "is this assembled site correct?" is the load-bearing wall. Build it early (Phase 3), not last.

**Working note:** internal agent communication lives in the Workspace at `/cms-admin` (Agents tab), backed by per-client ledger files and the `/api/agents/ledger` API.

---

## Architecture decision (2026-07-09): git-native, incremental Builder

Confirmed by Ridhi, supersedes the "regenerate from a config" model that Builder v1 proved:

- **Each client's website lives in its own GitHub repo**, not a local `builds/<slug>/` folder that gets deleted and regenerated. The repo IS the site; git history IS the project's daily progress log.
- **The Builder does not build from scratch.** For a website request, it clones/pulls the **existing template repo** (name TBD, Ridhi to provide) and builds on top of it per the client's spec, the template repo is the real starting point, not `templates/starter/` (which only proved the assembly mechanics on a placeholder).
- **Work incrementally, like a person would:** pull the current state of the client's repo → see what's already there, including any manual edits Ridhi made → make the specific change being asked for → commit → push. Never a blind wipe-and-regenerate. This is also what fixes the "Ridhi hand-edits something and it survives" problem: a manual edit is just a commit, and the next agent run starts from that commit, not from a fresh copy.
- **"Continue work each day until finished"**: an agent resuming a project just means pulling that client's repo, checking progress against the pipeline stage + ledger, and continuing. Same durable-state pattern already used for conversations (the ledger) and project stage (the pipeline), now extended to the actual code.
- **Consequence for QA (Phase 3):** the verification harness will need to handle whatever the real template's build/serve story is (this is unknown until the template repo is shared — could be a static build, could need `next build`/`next start`, etc.), not only the static-stub model `verify.mjs --manifest` was proven against.
- **Blocked on:** the template repo (name/access, Ridhi to provide — see Phase 1), and a GitHub API credential so agents can create/clone/commit/push per-client repos (see Phase 4). Builder v1's deterministic engine (module toggles, theme, fact-slot copy, QA handoff, escalation) is not wasted, it's the config layer that will sit on top of the git workflow once both land.

## Architecture decision (2026-07-13): deterministic assembly kept; visual variety via multiple template SETS

Confirmed by Ridhi, after explicitly considering and rejecting an AI-generates-code-per-client model:

- **Assembly stays deterministic.** The Builder remains config-in, known-output-out. This is deliberately protected because every safety mechanism already built depends on it: the fact-slot rule (agents never assert unsupplied facts), QA's reliability (it verifies a *bounded, known* output space), the change loop's revert safety net (re-run the same assembler, diff old-vs-new), and fixed per-module pricing. An AI freely writing per-client code would reopen all four. Off-menu/structural work remains the permanent human gate → Claude Cowork (Phase 0, unchanged).
- **Visual variety is solved by having many template SETS to choose from, not by non-deterministic building.** A "set" is a complete, visually distinct family: one website shell + one CMS admin core + one implementation of each feature module (catalog/careers/blog/gallery), all sharing a single strong visual identity, but technically identical in contract to every other set. A client picks a set; the same deterministic pipeline builds it.
- **How a set is chosen:** the intake form's existing "which visual direction fits you best?" question maps to a set (a deterministic *lookup*, not generation) instead of, as today, only a color preset inside one shared template. Today two clients who pick the same modules get the same layout with different accent colors; with sets they get genuinely different-looking sites.
- **The stitching contract:** each template (shell, CMS core, each module) ships a `manifest.json` whose `capability` field is STABLE across all sets (always `catalog`/`careers`/`blog`/`gallery`/`site`/`cms-core`) and whose `set` field ties a family together. An agent asks for "the catalog module for this client's set" via `capability` without knowing set internals. **Data shapes (Job/Article/Product/CatalogCategory/GalleryImage) are frozen identical across every set** so content-seeding keeps working unmodified. Route/nav ids stay identical in meaning across sets (`/catalog` everywhere).
- **Template vs functionality split (for outsourced set-building, e.g. ChatGPT Codex):** the set author builds the VISUAL + STRUCTURAL layer only — pages, components, admin editor UIs, forms, all UI states, typed data interfaces, and clearly-marked async *seams* with mock data — but NOT the real backend. Persistence, auth, email, and uploads are seams the Deneb4 assembler wires at stitch time. This is what let the **persistence decision (now resolved, see 2026-07-15 below: Turso via `d4-cms-core`) swap in without touching any set** — the set author stays persistence-agnostic; async seam signatures don't change when the network backend changes. Each set must run standalone on mock data with zero credentials so its design can be reviewed.
- **Non-negotiable for any new set** (or the existing QA harness auto-rejects the assembled site): passes axe-core with zero serious/critical violations incl. WCAG-AA contrast, no horizontal scroll at 375px, a designed empty state for every data view, renders with no env vars set, respects `prefers-reduced-motion`.
- **Our-side follow-up when a new set actually lands** (not blocking the set author): register it in `registry.json`, teach the intake→set mapping and the pricing engine that the set exists. Small, known work.

## Architecture decision (2026-07-15): client-site hosting is Vercel + Turso + Vercel Blob

Confirmed by Ridhi. Supersedes the earlier DigitalOcean-droplet-plus-local-JSON-files direction (that plan is dead; nothing was built against it besides the pricing analysis).

- **Why not a raw VPS (DigitalOcean/Hostinger Droplet):** correctly flagged as real ops toil that fights the zero-touch goal — manual Nginx, PM2, Certbot/SSL renewal, OS patching, an agent one Nginx-config typo away from bricking every client site on the box.
- **Why not plain Vercel/Cloud Run/DO App Platform (as first proposed as the fix):** all three are stateless — no persistent writable disk between deploys/instances — which breaks `d4-cms-core`'s filesystem-based JSON data store outright. That gap is real, not a caveat to hedge around.
- **The actual fix: Vercel (hosting/deploy) + Turso (data) + Vercel Blob (uploads).** Turso is libSQL — a real database with an HTTP driver serverless functions can use, and (unlike Supabase's free tier) built on always-on edge replicas with no pause-after-inactivity model. Vercel Blob replaces local file writes for uploaded images/PDFs. Together they give an actually-stateful app on an actually-stateless host, with zero server to ever patch, SSH into, or restart.
- **Cost:** Vercel Pro $20/mo flat (Hobby is non-commercial-use only). Turso free tier (100 DBs, 500M row reads/mo, 10M row writes/mo, 5GB storage) comfortably covers dozens of low-traffic B2B client sites before any usage-based cost kicks in. Vercel Blob: 1GB free, then $0.023/GB. Realistic total today: **$20/month**, cheaper than the droplet plan it replaces, with no ops burden at all.
- **Done (2026-07-15):** `d4-cms-core`'s `data-store.ts` rewritten from sync fs-JSON to async libSQL (Turso in production via `TURSO_DATABASE_URL`/`TURSO_AUTH_TOKEN`, a local SQLite file in dev with zero credentials needed — same code path both ways); the upload route rewritten to use Vercel Blob when `BLOB_READ_WRITE_TOKEN` is set, falling back to local disk in dev. All four content modules (catalog/careers/insights/gallery) and their call sites converted to the now-async data layer. `d4-site-template`'s contact-form fallback migrated off raw fs the same way. Verified end-to-end: assembled a throwaway 5-module client, `npm install`, clean `tsc --noEmit`, clean `next build`, live server on the local SQLite fallback answering real reads (`/api/careers`) and writes (`/api/careers/apply`) that persisted to `data/cms.db`.
- **Feature gating for clients who don't want the full module set:** already solved — the existing `modules[]` array in each build config plus `registry.json`'s closed menu means an unwanted module (e.g. careers, gallery) is simply never assembled into that client's build. No runtime toggle layer needed or planned; a client never receives code for a feature they didn't select.
- **Still open, [Ridhi]:** create a Vercel account + project linked to the `deneb4admin` GitHub org (or wherever client repos live), a Turso account + one database + auth token, and enable a Blob store on the Vercel project — needed before any real (non-dev-fallback) client deploy, not before now.
